# Content Production Workflow

## 1. Purpose

This document defines the complete content production pipeline for the HSC-ai platform. It covers how questions progress through their lifecycle from creation to publication, the review process, quality tracking, and how published content integrates with curriculum coverage.

All content on this platform directly affects the learning outcomes of children preparing for NSW exams. A structured, auditable production workflow is essential for quality assurance.

---

## 2. Question Lifecycle

```
┌──────────────────────────────────────────────────────────────┐
│                    QUESTION LIFECYCLE                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   [Manual / OCR / AI / Imported]                             │
│          │                                                   │
│          ▼                                                   │
│       DRAFT                                                  │
│   Content created; not visible to students.                  │
│   Admin may edit freely. quality_score optional.             │
│          │                                                   │
│          ▼ (submit-review)                                   │
│       REVIEW                                                 │
│   Admin has submitted for review.                            │
│   quality_score and review_notes may be set.                 │
│          │                │                                  │
│          ▼                ▼                                  │
│     APPROVED          REJECTED                               │
│   reviewed_by_admin   reviewed_at set                        │
│   reviewed_at set     → can return to draft                  │
│          │                                                   │
│          ▼ (publish)                                         │
│     PUBLISHED                                                │
│   published_at set. Visible in:                              │
│     - curriculum coverage                                    │
│     - exam instances                                         │
│     - question pools                                         │
│     - student analytics                                      │
│          │                                                   │
│          ▼ (archive)                                         │
│     ARCHIVED                                                 │
│   Excluded from new exams.                                   │
│   Historical attempts preserved.                             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### State Transition Rules

| From | To | Condition |
|---|---|---|
| `draft` | `review` | Admin submits for review |
| `review` | `approved` | Admin approves; reviewed_by_admin_id and reviewed_at set |
| `review` | `rejected` | Via existing `transition_status` API |
| `rejected` | `draft` | Admin revises and resubmits |
| `approved` | `published` | published_at set |
| `published` | `archived` | Admin decision |
| `approved` | `archived` | Admin decision |
| `rejected` | `archived` | Admin decision |

## 3. Content Source Tracking

Each question carries a `source_type`:

| Value | Description |
|---|---|
| `manual` | Written directly by an admin |
| `ai` | Generated by AI under admin direction |
| `ocr` | Extracted from scanned/PDF documents |
| `imported` | Imported from external sources (new in M4.7C) |

Source distribution is visible on the Content Review dashboard and Curriculum Dashboard.

---

## 4. Quality Review

The Question model includes review-specific fields added in M4.7C:

| Field | Type | Description |
|---|---|---|
| `quality_score` | SmallInt 1-5 | Admin-assigned quality rating during review |
| `review_notes` | Text | Reviewer comments |
| `reviewed_by_admin_id` | FK → admin_profiles | Admin who approved/rejected the question |
| `reviewed_at` | DateTime | When the review decision was made |
| `published_at` | DateTime | When the question was published |

These fields provide a traceable audit trail for every published question.

## 5. Review Queue

The admin review page (`/admin/content/review`) provides:

- **Filters**: by status (draft/review/approved), source type, subject, exam type
- **Status badges**: color-coded per lifecycle state
- **Individual actions**: Submit for Review, Approve, Publish, Archive
- **Bulk actions**: Select multiple questions and approve/publish/archive
- **Stats**: Draft count, In Review count, Approved count, Published count, Published This Week/Month

### API Endpoints

| Method | Path | Purpose |
|---|---|---|
| GET | `/admin/content/review` | List review queue (with filters) |
| POST | `/admin/content/questions/{id}/submit-review` | Submit draft for review |
| POST | `/admin/content/questions/{id}/approve` | Approve review question |
| POST | `/admin/content/questions/{id}/publish` | Publish approved question |
| POST | `/admin/content/questions/{id}/archive` | Archive a question |
| POST | `/admin/content/bulk-action` | Bulk approve/publish/archive |
| GET | `/admin/content/stats` | Content statistics |

## 6. Coverage Integration

Curriculum coverage counts only `status = published` questions. Drafts, review, and approved-but-unpublished questions do not contribute to coverage percentages.

This ensures the coverage dashboard reflects real, student-facing content — not work in progress.

---

## 7. Audit Trail

Every published question has:

- **Creator**: `created_by_admin_id` — who wrote it
- **Reviewer**: `reviewed_by_admin_id` — who approved it
- **Review timestamp**: `reviewed_at`
- **Publication timestamp**: `published_at`

Future: Review history log for tracking rejections and resubmissions.

---

## 8. Dashboard Integration

The Content Review page shows:
- Draft / In Review / Approved / Published counts
- Published this week / month
- Total question count

The Curriculum Dashboard page integrates content stats for coverage-aware decision making.

---

## 9. Best Practices

1. **Two-person review**: Where possible, the creating admin should not be the reviewing admin. In V1 single-admin setups this is relaxed but logged.
2. **Quality first**: Set `quality_score` during review. Questions with score < 3 should be revised before publication.
3. **Source awareness**: Track source distribution on the dashboard. Aim for >50% original/manual content.
4. **Coverage-driven seeding**: Use the curriculum dashboard to identify uncovered outcomes, then create and publish questions targeting those gaps.
5. **Archive, don't delete**: Archiving preserves historical attempt data. Deleting would break referential integrity.
